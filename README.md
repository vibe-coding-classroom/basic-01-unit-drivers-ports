針對 **`basic-01-unit-drivers-ports` (驅動程式與通訊埠)** 單元，這是硬體開發的最前哨站。它要求學員克服開發環境與實體硬體之間的第一道高牆：「連線穩定度」。透過對 **波特率 (Baud Rate)** 的理解與 **Loopback (環回測試)** 的實作，學員能學會如何科學地排除硬體通聯故障。

以下是在 **GitHub Classroom** 部署其作業 (Assignments) 的具體建議：

### 1. 範本倉庫 (Template Repo) 配置建議
初學者最常卡在「沒反應」，因此範本應提供明確的引導結構，建議包含：
*   **📂 `platformio.ini`**：**配置練習對象**。故意遺漏 `monitor_speed` 設定，要求學員手動補全為 `115200` 以解決亂碼。
*   **📂 `src/main.cpp`**：基礎 Hello World 程式。每隔 1 秒從 Serial 埠輸出一次當前運行秒數，供學員測試連線。
*   **📂 `docs/DIAGNOSTIC_REPORT.md`**：通聯診斷報告。要求學員記錄：`作業系統版本 | 橋接晶片型號 | 分配到的 COM/tty 埠號 | 是否通過 Loopback 測試`。
*   **📂 `assets/`**：圖片存放區。學員需在此存放「裝置管理員/終端機偵測圖」與「順利接收到的 Serial Data」截圖。

---

### 2. 作業任務部署細節

#### 任務 1：環境通聯與偵錯 (Driver & Hardware Debugging)
*   **目標**：確保學生能正確識別硬體「身分」，並排除線材或驅動的初階問題。
*   **Classroom 部署建議**：
    *   **埠號識別**：學員需在 README 中回報其電腦偵測到的埠號（如 Windows 的 `COM3` 或 Mac 的 `/dev/tty.usbserial-xxx`）。
    *   **晶片查核**：要求學員標註其開發板使用的橋接晶片（CP2102 或 CH340），並證明已安裝對應驅動。
    *   **驗證要點**：提交一張截圖，證明系統已成功載入該通訊埠。

#### 任務 2：亂碼解密實驗 (Baud Rate Synchronization Lab)
*   **目標**：透過「故意製造錯誤」，讓學生深刻理解連線語速同步的重要性。
*   **Classroom 部署建議**：
    *   **故意失步**：在 `platformio.ini` 中故意將監測速率設為 `9600`，但程式內使用 `115200`。
    *   **亂碼記錄**：錄製或截圖此時出現的「亂碼」現象，並解釋其成因。
    *   **同步修復**：修正設定並展示清晰的原文輸出，完成「修復任務」。
    *   **PR 提交內容**：提交修正過的 `platformio.ini`。

#### 任務 3：Loopback 自我對話測試 (Hardware Loopback Test)
*   **目標**：掌握硬體界最強大的「隔離問題法」，判斷故障是在電腦側、線材側還是單晶片側。
*   **Classroom 部署建議**：
    *   **物理短路操作**：學員需使用杜邦線短路板子上的 `TX` 與 `RX` 引腳。
    *   **回音測試**：開啟序列監視器並輸入文字，記錄文字是否成功「反射」回螢幕（Echo Test）。
    *   **診斷結論撰寫**：在 `README.md` 中詳細說明：此測試成功代表從「電腦到橋接晶片」這條鏈路是健康的。
    *   **最終 Push**：整理所有實驗結果與截圖後進行最終提交。

### 3. 硬體通訊初學者導師點評標準 (Connectivity & Support Standards)
此單元的價值在於 **「對硬體通聯邏輯鏈的清晰認知」**：
*   [ ] **排障系統度**：學員是否能依據「線材 ➡ 驅動 ➡ 埠號 ➡ 速率」的順序邏輯性地描述問題？
*   [ ] **參數配置正確性**：`monitor_speed` 是否與程式碼定義嚴格對齊？
*   [ ] **專業術語運用**：是否能正確使用 `UART`, `Baud Rate`, `USB-to-Serial` 等專業詞彙描述作業過程？
